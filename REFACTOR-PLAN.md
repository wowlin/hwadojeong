# 리팩토링 계획 — 화도정 3D 모델 코드

작성일: 2026-07-15 · 갱신: 2026-07-15(두 세션 계획 병합) · 대상: `src/` 전체 + 검증 게이트 + 죽은 산출물 · 원칙: **화면(렌더 결과)은 1픽셀도 안 바뀐다**

## 1. 목표 (요청 5가지)

1. 프로젝트 코드 전체 검사
2. 이후 코딩이 편리하도록 구조 개선
3. 너무 큰 파일 제거 — 편집·검색 시간 단축
4. 매직넘버 최소화 — 기준값에서 파생시켜, 기준 하나 고치면 관련 값이 전부 따라오게
5. 반복 부품(콘센트·실링팬·의자 등) 공통 모듈화 — 같은 부품은 한 벌만 정의하고 가져다 쓴다

### 2-2. 발견된 문제 (검사 결과 — 전 항목 코드 실물 재검증 완료)

**A. 거대 단일 파일** — 어떤 편집이든 3,909줄에서 검색해야 함. 편집 왕복 시간의 주범.

**B. "스코프 밖이라 값 고정" 하드코딩** — 한 파일인데도 블록 스코프에 갇혀, 이미 계산된 값을 숫자로 다시 박은 곳들. 코드 스스로 주석으로 자백하고 있음:
- 실외기실 벽 위치 `0.40`, `1.80` — 실제로는 「게스트룸2 뒤끝」「화장실 앞벽」인데 숫자 고정 (주석: "스코프 밖이라 값 고정")
- 홈리프트 안방쪽 면을 s2 외벽 코드가 `(외벽+계단폭)+9단+1.2+1.5`로 **재계산** — 계단 코드가 이미 계산한 값의 사본
- 복도창 X중앙 `리프트면 − 2.025`가 **5곳**에 복제 (1층·2층·3층 콘센트·창)
- `기초윗면+슬래브두께`(1층 바닥 윗면), `2·3층 바닥 레벨`, `천장 밑면` 재계산 5곳+
- 에어컨 위치: 설치 코드 `창뒤끝(+2.2)+이격 0.3` vs 콘센트 쪽 `+2.5` — **재검증 결과 두 값은 현재 동일**(2.2+0.3=2.5). "이미 어긋난 사본"은 오진. 다만 같은 값을 두 곳이 따로 계산하는 사본 구조라 창이 움직이면 그때 어긋남 → 일반 단일 출처화 대상(§4-#10).

**C. 중복 정의(단일 출처 위반) — 요청 5(공통 부품)의 대상**
- 콘센트 그리기 헬퍼가 **5벌** (s1 `outlet`, s2 `wallOutlet`, `inOutlet` 2벌, `inHeatOutlet`) + 외부 콘센트 인라인 2곳
- 미서기 유리 재질(고정짝/미닫이짝 색)이 **6곳+**에서 매번 `new Material(...)`로 재생성
- 폴딩 프레임(다크그레이) 재질 3곳 재생성
- 평면 기준선(회청 보조선) 그리기 코드가 s1·s2 두 벌
- `MAT_H`는 `matFoundationH`의 단순 별칭, `loftFloorThickness`는 `secondFloorThickness`의 별칭
- 실링팬·전등·의자·테이블 등 반복 소품도 사용처마다 다시 그림 → fixtures 부품 라이브러리로 통일

**D. 실행 순서 의존 재질 주입** — `materials.roofInsul`·`snowGuard`·`gutter`를 **s1 지붕 블록이 실행되면서** materials 객체에 주입(main.js 961·979·997행 확인). s2 지붕이 이걸 몰래 빌려 씀 → s1 지붕 코드를 지우면 s2 지붕이 깨지는 숨은 결합.

**E. 죽은 코드**
- 버튼 높이 통일 함수가 존재하지 않는 셀렉터(`.controls button`)를 조회 — 확인 결과 그 클래스는 화면에 0건, 항상 no-op
- `stairWallObjects`는 비어 있는 채 숨김 처리만 반복 (주석에도 "미사용" 명시)

**F. 검증 게이트 드리프트** — `npm test` ④가 돌리는 `scripts/audit-layout.mjs`(452줄)가 **src를 전혀 읽지 않고(import 0건) 자기 사본 상수로만 검사**. 그 사본이 이미 옛 설계(집폭 8.5·기초 0.5·외벽 0.2 — 현행 8.2·0.35·0.3)라, 사실상 **현재 모델을 검사하지 않는 게이트**.

**G. 구역 위반 잠복** — S2 배너 구역 안에 데크 계단틀·화목난로 그리기가 있고, 격리 테스트 ⑭의 감시 목록에 해당 그룹들이 빠져 있어 통과해 옴. 그룹 귀속(s1 부재인지 s2 자체 부재인지)은 분할 단계에서 파일 경계로 확정·해소.

**H. 좌표성 리터럴 전수 분포 (신규)** — main.js 소수 리터럴 **2,364개**(s1 809 · s2 1,424 · UI/공유 131). §4의 "알려진 사본" 목록 밖에도 절대좌표가 본문 전반에 박혀 있어, 목록 수리만으로는 "기준 하나 고치면 전부 따라오게"가 안 됨 → 전수 상대값화 단계(§5-8) 필요.

**I. 죽은 산출물 파일 (신규)** — CLAUDE.md §1이 금지한 임시 캡처 스크립트가 scripts/에 10여 개 잔존(`shot-*-tmp.mjs`·용도 끝난 shot-* 다수), 어디서도 참조 안 되는 `stop-gate.sh`(94줄), 루트에 캡처 png 29장(git 추적 8·미추적 21).

### 2-3. 구조적 제약 (리팩토링 시 반드시 보존)

- **scene 추가 순서가 곧 토글 그룹 소속**: `captureInto`·`_start = scene.children.length` 슬라이스 패턴이라, 부재 생성 순서를 바꾸면 화면 토글이 깨진다 → 모든 빌드 호출 순서를 원본 그대로 유지.
- 파일 내용을 새로 타이핑하지 않고 **줄 범위를 그대로 이동**(shell 이동)해 치수 오타 가능성을 0으로.

### 4-2. 전수 상대값화 (§5-8단계 — 신규)

목록 밖 좌표성 리터럴 2,364개 전체를 구역별로 훑어 교체한다:
- **좌표** → 기준 앵커(buildingW·insideX0/Z0·firstFloorY·s2Geo 레벨 등) ± 파생식

## 6. 동작 불변 보장

- **스냅샷 가드가 기계 증명**: 매 단계 전체 부재 덤프 diff=0. 좌표를 파생식으로 바꿀 때 생기는 이진 부동소수 오차는 1e-6 반올림이 흡수(실제 오차 ~1e-15) — 순서 보존·육안에 의존하지 않는다. 특히 7·8단계(값을 식으로 다시 쓰는 단계)에서 필수.
- 분할 단계(2~5)는 코드를 **줄 범위 그대로 이동**(재타이핑 금지) → 치수 오타 원천 차단
- 부재 생성(scene 추가) **호출 순서를 원본과 동일**하게 유지 → 토글 그룹 소속 불변
- 매 단계: `npm test`(문법·빌드·회귀 17종) + `npm run lint`(미사용 import 0)
- 모듈 분리로 빠진 import는 eslint `no-undef`가 전수 검출
- 마지막에 `npm run check:render` 1회(런타임 에러 0·캔버스 정상 — 기존 자동 게이트, 시각 판단은 사용자 몫 유지)
- 시각 최종 확인: 사용자가 s1·s2 탭에서 토글별 확인
