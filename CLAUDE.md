# CLAUDE.md — 작업 규칙 (화도정)

이 규칙은 매 세션 강제 적용. 위반 = 사용자 시간 낭비.

## 0-0. 무조건 간결·명확 (모든 세션·프로젝트 전체 기본)
- **질문엔 핵심 답 한 줄.** 복창·배경·군더더기·자기변명·"참고로" 금지.
- 안 물어본 것 덧붙이지 마라. 제안은 꼭 필요할 때만 맨 뒤 1줄.
- 단정할 수 있으면 단정. caveat로 답 흐리지 마라.

## 0-1. 빠르게 — 왕복 최소화
- **느림의 원인은 검색·빌드·렌더의 왕복 횟수다.** 한 번에 하나씩 grep/read 하지 말고, 필요한 정보를 **한 번의 호출에 몰아서** 모은다(여러 grep을 한 명령에, 의존 없는 도구 호출은 한 응답에 병렬로).
- 코드 위치는 **추측 말고 한 번에 넓게 검색**해 좌표·상수·재질을 함께 확보한 뒤 곧장 수정한다.
- **빌드·렌더(`shot`)는 작업당 1회**가 원칙. 수정 전 여러 번 미리 돌리지 마라. 검증은 `npm test` → 필요한 화면만 1회 렌더.
- 같은 작업에 같은 검색을 반복하지 마라. 이미 읽은 값은 다시 안 읽는다.

## 0-2. 구조적 합리성 > 편집 편의 (설계 결정의 최우선 기준)
- **부재를 추가·수정하는 방법이 여럿이면, '내 코드가 덜 바뀌는 쪽'이 아니라 '실제 건물이 그렇게 서는 게 맞는 쪽'을 택한다.** 편집 편의·회귀 위험 회피·수정 범위 최소화를 설계 판단 기준으로 삼지 마라. 정답이 여러 곳을 손대는 큰 수정이면, 그 큰 수정을 한다.
- **이미 있는 부재(기초·골조·벽 등)를 활용할 수 있으면 새로 만들지 말고 그것에 맞춘다.** 온통기초가 있는데 독립기초를 덧대는 식의 중복·낭비 부재는 금지 — 이미 있는 것을 넓히거나, 부재 위치를 이미 있는 것 위로 옮겨 맞춘다.
- 결정 전 스스로 물어라: "이게 내 코드가 편해서 고른 방법인가, 건물이 그래야 맞아서 고른 방법인가?" 앞이면 멈추고 뒤를 택한다.

## 0. 시킨 것만, 딱 그만큼
- 요청한 것 외에 **아무것도 바꾸지 마라.** "색만" = 색만(크기·두께·위치 그대로). "위치만" = 위치만.
- 추측으로 **추가 작업 금지.** 애매하면 1줄로 묻고 멈춰라. 추측해서 더 하지 마라.
- 사용자가 한 단계씩 지시하면 **한 번에 하나만** 처리. 묶지 마라.
- 시키지 않은 파일(특히 `src/` 모델 코드) 건드리지 마라.
- 색상 변경 = 해당 `materials.xxx`의 **color 값 한 토큰만** 수정. 줄 재작성·그룹 이동·구조 변경 금지.

## 1. 검증 — 시각 확인은 사용자 몫
- `src/` 모델 코드 편집 후 **`npm test`(문법·빌드)만** 돌린다. 통과 못 하면 보고 말고 고쳐라.
- **렌더/시각 확인은 절대 내가 하지 마라.** `npm run shot`·`shot:side`·캡처·이미지 열기 금지. 위치·정렬·기하가 맞는지는 **사용자가 직접 화면으로 확인**한다.
- 임시 캡처 스크립트(`_tmp-*.mjs` 등) **절대 만들지 마라.**
- 화면 결과를 내가 "맞다/됐다"고 단정하지 마라 — 시각 판단은 사용자에게 넘긴다.

## 1-1. 확인 요청 전에 '사용자가 보는 곳'에 반영 (예외 없음)
- **사용자는 dev 서버(`http://127.0.0.1:5173/`)로 본다. 그 서버는 항상 LIVE 체크아웃(`/Users/aine/work/three-house`)의 현재 브랜치·작업 트리를 서빙한다.** 워크트리·다른 브랜치에만 있는 편집은 **사용자에겐 없는 것**이다.
- **"확인해 주세요"라고 끝내기 전에 반드시 LIVE에 반영한다.** 워크트리에서 작업했으면: 커밋 → `git -C <워크트리> rebase <LIVE 브랜치>` → `git -C /Users/aine/work/three-house merge --ff-only <브랜치>`. 반영은 **묻지 말고 즉시**(§3 자동 커밋과 같은 취지) — "머지할까요?"·"PR 올릴까요?" 금지.
- **반영했으면 서빙까지 확인하고 보고한다**: `curl -s http://127.0.0.1:5173/src/<파일> | grep <새 식별자>`. 이건 렌더/시각 확인이 아니라 **배달 확인**이라 §1의 'shot 금지'와 무관하다 — 화면 판단은 여전히 사용자 몫.
- LIVE가 앞서갔으면 rebase로 따라붙는다. LIVE의 미커밋 작업 파일과 내 파일이 겹치면 **덮지 말고** 그 사실을 알리고 멈춘다.
- **`git reset --hard`·`checkout --`로 남의(그리고 내) 미커밋 편집을 날리지 마라.** 게이트 시험 등으로 되돌릴 일이 있으면 **먼저 커밋해 안전지대를 만든 뒤** 시험한다(실제로 이 규칙 문서가 그렇게 한 번 날아갔다).
- 강제: Stop 게이트 `scripts/applied-gate.sh` — 워크트리의 src/ 변경이 LIVE 브랜치에 없거나 미커밋으로 남아 있으면 종료를 차단한다.

## 2. 보고
- **모든 응답은 한글.** 코드·식별자·명령어 제외.
- 질문엔 **한 줄로 단정해서 먼저 답**하라. 요청 복창 금지. 번호별 요청은 번호별로 보고.
- **답을 단서(caveat)로 흐리지 마라.** 확신하면 "네/아니요"로 단정한다. 추가·제안은 **필요할 때만 답 뒤에 짧게 분리**하고, **추가가 답보다 길면 안 된다.** 안 물어본 것(아직 안 한 일·다른 제안)으로 본문을 채우지 마라.
- 안 된 걸 된 척 금지. 확인 못 한 건 "확인 못 했다"고 명시.
- **책임 회피 금지.** 이 세션에서 내가 만들거나 고친 코드를 "기존 코드"라고 떠넘기지 마라. 작성자가 의심되면 `git blame`·`git log -S`로 먼저 확인하고, 내 산출물이면 인정하고 고친다.
- **코드로 답하지 마라. 화면으로 답해라.** 사용자는 코드를 안 본다. 줄번호·변수명·파일경로·함수명을 답에 쓰지 마라(코드 확인은 내가 하고, 답변엔 안 쓴다). 답은 화면(렌더된 모델에 보이는 것)으로: 위치는 도면 방향(앞/뒤·좌우·주방쪽/안방쪽), 크기·치수는 m 단위로. 내부 구현이 원인이어도 사용자에겐 화면에 보이는 결과로 설명한다.

## 4. 하네스 (생성-검증) — 단일 서브에이전트 모드
**목표:** src/ 모델 코드 편집을 "편집→(선택)검증→커밋"으로 안전하게, 왕복 최소로 처리.

**모드:** 에이전트 **팀 아님.** 단일 파일·순차 편집이라 팀 통신 오버헤드가 이득보다 큼(§0-1) → 생성-검증 **단일 서브에이전트(verifier)**. 오케스트레이터 스킬 = `model-edit`.

**트리거:** src/ 모델 코드 편집·"화면: 요청" → `model-edit` 스킬. 단순 질문·추천은 직접 응답. 파일 지도: s1=src/s1/* · s2=src/s2/* · 공유 헬퍼=src/*.js · main.js=오케스트레이터(파이프라인 호출만).

- 검증은 **§1이 단일 출처(`npm test`).** verifier 에이전트는 **범위 위험 편집에서만 선택 호출** — 값·파라미터 분기, 공용 함수 수정처럼 의도 밖 부재까지 바뀔 수 있을 때. 단순·국소 편집은 `npm test`만.

## 5. 프로그램 구조 — 단일 출처·누적 (앞→뒤)
- **설계 파이프라인(배치→기초→바닥틀→바닥→계단→1층→다락→지붕)이 '진짜 집'이자 단일 출처다.** 각 부재는 처음 나오는 단계에서 **딱 한 번만** 정의한다. 뒤 단계는 그것을 자동 누적하고, 거기서 사용자가 할 일은 **'보이기/숨기기'뿐.** 같은 부재를 뒤 단계에서 다시 그리거나 좌표를 다시 잡지 마라.
- **참고 토글(현행 버튼: s1 = 지붕·태양광 / 다락 바닥·내벽·외벽 / 포치 데크·폴딩·지붕·프레임 / 1층 외벽·콘센트·천장·화장실·계단·바닥 / 기초·측백담장·옆집담장)은 '부품 빠뜨림 방지 체크리스트'다. 위치용이 아니다.** 건드리지 말고, 필요한 부품이 있으면 거기서 **복사해** 파이프라인에 넣는다.
- **뒤 단계에서 앞 단계 산출이 안 따라오면 사용자 문제가 아니라 '단일 출처로 안 묶인 내 코드' 문제다.** 설계 작업의 일부로 조용히 단일 출처로 묶어라. "적용 안 됐으니 해달라"고 사용자가 다시 부르게 하지 마라. **사용자는 설계만, 구조 코드는 내가.**
- 새 설계는 **항상 단일 출처로 구현**해 자동 누적되게 한다. "됐다" 전에 **다락·지붕 단계까지 렌더로 확인**한다.
- 한 곳만 고치고 "끝"이라 하지 마라 — 같은 부류(형제) 중복을 **스스로 전수 점검**해 함께 정리한다.

## 5-1. 콘센트 = 각 층 '외벽' 그룹 (예외 없음)
- **앞으로 추가할 모든 층의 모든 콘센트(내부·외부 불문)는 반드시 그 층의 '외벽' 메뉴 그룹에 넣는다.** s2 1층→`s2Wall1Objects`(bF1Wall), 2층→`s2Wall2Objects`(bF2Wall), 3층→`s2Wall3Objects`(bF3Wall). 싱크대·가구 등 다른 토글에 넣지 마라.
- `captureInto(s2Wall<층>Objects, …)` 블록에 그린다. 좌표 재계산 시 이중 수집 금지(한 그룹에만).

## 5-2. 메모 숫자 = 코드 연동 (예외 없음)
- **메모(NOTES)에 적는 모든 치수·수치는 코드 변수·상수에서 계산해 `${...}`로 넣는다.** 숫자 리터럴에 단위(m·mm·㎡ 등)를 붙여 박아두지 마라 — 코드가 바뀌면 메모도 자동으로 바뀌어야 한다.
- 불가피한 규정값(법정 이격·건폐율 한도 등 모델 부재가 아닌 값)만 예외. 그때도 `test/smoke.test.js` ⑯의 `ALLOW`에 **근거 주석과 함께** 등록해야 통과한다.
- 강제: `npm test` ⑯ '메모 숫자 = 코드 연동'이 NOTES 블록을 스캔해 하드코딩 치수를 차단한다.

## 5-3. 메모 서식 = 보기좋게 (예외 없음)
- **메모(NOTES)는 한 줄에 여러 수치를 `·`로 죽 이어 붙이지 마라.** 항목이 여럿이면 `［구획］` 머리글로 묶고, 각 수치는 **한 줄에 하나씩 라벨 정렬**해 적는다(s2StairF1·s2Floor1 형식).
- 구획 사이는 빈 줄(`''`)로 띄운다. 라벨은 값이 세로로 정렬되게 뒤에 공백을 맞춘다.
- 강제: `npm test` ⑰ '메모 서식'이 한 줄에 수치 `·`를 3개 이상 몰아 쓴 run-on 라인을 차단한다.

## 6. 코드 품질 — 게이트로 강제 (세션·모델 무관)
- **편집 후 `npm run lint`(eslint)를 반드시 돌려 0으로 유지.** 미사용 import·변수·죽은 코드(uncalled 함수 포함)를 남기지 마라. Stop 게이트 `scripts/lint-gate.sh`가 eslint 에러 시 종료를 차단한다.
- **주석에 매직넘버 금지.** 코드가 계산·정의하는 치수(좌표·높이·너비·층고 등)를 주석에 절대숫자로 복제하지 마라 — 변수명·파생식으로 쓴다. 새 숫자로 교체하지 말고 숫자 자체를 걷어낸다. (예외: 상수 정의부의 자기 값, 실제 제품 스펙.)
- **나쁜 패턴은 지적받기 전에 먼저 알리고 고쳐라.** 죽은 코드·중복·단일출처 위반을 발견하면 요청 없이 정리하고 보고. 검사 가능한 부류(미사용 export·죽은 참조 등)는 grep 어림짐작 말고 **린터·전수 검사로 0을 확정**한다.
- **금지어 '거실'**: 低X 앞공간은 '주방'. Stop 게이트 `scripts/no-geosil.sh`가 응답에 '거실'이 있으면 종료를 차단한다.

## 도면 방향
- 입구·데크 = 아래(앞). 평면 좌 = 안방(高 x). 평면 우 = 주방(低 x).
